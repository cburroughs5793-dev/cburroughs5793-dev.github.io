How I Debug RTK GPS Issues on Deployed Robots

By Christopher Burroughs · February 2026 · 6 min read

RTK GPS is supposed to give you centimeter-level accuracy. In reality, you spend half your time figuring out why it's giving you garbage data.

I've spent the last few years supporting autonomous line-marking robots in the field. These machines need precise positioning to paint straight lines on sports fields — we're talking 2-3 cm accuracy. When RTK works, it's beautiful. When it doesn't, you're troubleshooting at 6 AM before a soccer game.

Here's the process I've developed for diagnosing RTK GPS issues on deployed robots.

The First Question: Is It Actually a GPS Problem?

Before diving into GPS debugging, rule out the obvious. I've seen "GPS issues" that turned out to be:

  • Loose antenna cables
  • Motor interference (poor shielding)
  • Wrong coordinate frame in the navigation stack
  • Base station configuration drift

My first step is always checking the basics in RViz. Is the robot's position jumping around, or is it just offset? Jumping usually means GPS. Consistent offset usually means transforms or calibration.

Step 1: Check Fix Status

RTK GPS reports a "fix type" that tells you how good your solution is:

  • No fix / Single — Basically useless for precision work (±2-5 meters)
  • Float — Getting corrections, but not locked (±0.5-1 meter)
  • Fix (RTK Fixed) — This is what you want (±2 cm)
# Check GPS status via ROS topic
rostopic echo /gps/fix_type

# Or check NMEA sentences directly
# Look for GGA message, field 6:
# 4 = RTK Fixed, 5 = RTK Float, 1 = Single

If you're stuck in Float or Single, the problem is usually corrections — either they're not arriving, or they're arriving too slowly.

Step 2: Verify Correction Stream

RTK needs a base station sending corrections. Most issues I see fall into three categories:

1. Corrections Not Arriving

  • Check NTRIP connection status
  • Verify network connectivity (cellular signal, WiFi range)
  • Confirm base station is online and broadcasting

2. Corrections Too Old

RTK corrections have a shelf life. If latency exceeds ~5-10 seconds, accuracy degrades. I check the "age of corrections" field in the GPS output.

# Age of differential (seconds) - should be < 5
rostopic echo /gps/age_of_differential

3. Wrong Base Station

If the robot is receiving corrections from a base station that's too far away (>30 km), accuracy suffers. I've seen this when someone accidentally connects to a public NTRIP caster with the wrong mountpoint.

Step 3: Check the Physical Setup

GPS is surprisingly sensitive to physical factors:

Common culprits I've found: Antenna mounted near metal (reflections), antenna cable damaged during transport, antenna not level, obstructions blocking sky view (trees, buildings, the robot's own frame).

A good test: put the robot in an open field with clear sky view. If RTK works there but not at the deployment site, you have an environmental problem.

Step 4: Visualize in RViz

I always pull up RViz to see what the navigation stack is actually seeing:

  • GPS position topic — Is it updating? How often?
  • Covariance — High covariance = GPS isn't confident
  • TF transforms — Is the GPS frame correctly connected to base_link?

One trick: plot the raw GPS path over time. If you see sudden jumps or drift patterns, that tells you whether it's a fix loss, multipath interference, or transform issue.

Step 5: The Log Dive

When all else fails, I pull logs and look for patterns:

  • Does the issue happen at a specific time of day? (GPS satellite geometry changes)
  • Does it correlate with robot actions? (Motor noise, orientation)
  • Is it location-specific? (Environmental interference)

I've solved problems by discovering that RTK dropped every time the robot turned on its paint sprayer (electrical interference) and every day at 3 PM (satellite geometry window).

The Takeaway

RTK GPS debugging is systematic. Start broad (is it even a GPS problem?), then narrow down (fix type → corrections → physical → environment). Keep notes on what you find — patterns emerge over deployments.

The robots I support now have way better uptime because we documented every failure mode and built checks into the startup routine. Boring? Yes. Effective? Absolutely.
